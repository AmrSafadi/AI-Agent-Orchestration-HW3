# Project Blueprint

This document is the single source of truth for the project vision. It is written for three readers:

- a course grader who needs to evaluate alignment with the assignment,
- a teammate who needs to understand the system before coding,
- another AI assistant that may continue implementation later.

## 1. Project Motivation

The project demonstrates how a broad generative task can be decomposed into an organized multi-agent workflow. Instead of asking one model to "write a book and make a PDF", the system separates reasoning-heavy work from deterministic engineering work.

The intended final product is a professional article/book generated by a CrewAI team and rendered as a LaTeX PDF. The project also demonstrates production-minded ideas from the course: clear agent roles, task context, sequential orchestration, validation, observability, cost awareness, and reliable artifact generation.

The generated document is **primarily Hebrew (RTL)**. `templates/latex/main.tex.j2` sets `\setmainlanguage{hebrew}`, `\setotherlanguage{english}`, and `\setmainfont{David CLM}`. The current submission manuscript is a 7-chapter football analytics document (`data/intermediate/sample_book_plan.json`), with English kept inline only for technical terms. `templates/latex/chapter.tex.j2` includes an explicit `\begin{english}` LTR block that demonstrates the RTL-to-LTR BiDi transition.

## 2. Assignment Requirements

The final project must produce a professional PDF that includes:

- Multiple specialized agents

- Professional PDF generation
- Cover page
- Table of contents
- Chapters and sections
- Bibliography and citations
- Images

Version 1 uses a sequential CrewAI process and exactly five agents:

- Planner Agent
- Research Agent

Citation management, graph generation, validation, and PDF compilation remain deterministic Python components.

## 4. Final System Architecture

The final system has two major layers:

1. CrewAI reasoning layer
2. Deterministic Python harness layer

```text
Project Config
  -> Sequential CrewAI Crew
     -> Planner Agent
     -> Research Agent
     -> Writer Agent
     -> Reviewer Agent
     -> LaTeX Agent
  -> Structured Artifacts
  -> Deterministic Harness
     -> CitationManager
     -> GraphGenerator
     -> Validator
     -> LaTeX Renderer
     -> PDFCompiler
  -> Final PDF
```

## 5. Agent Responsibilities

### Planner Agent

Creates the document blueprint: title, audience, chapter plan, section plan, required feature placement, page estimate, and acceptance checklist.

### Research Agent

Produces a research pack from the plan: key concepts, terminology, source candidates, chapter notes, and unsupported-claim warnings.

### Writer Agent

Writes the manuscript from the plan and research context. It includes placeholders for citations, image, graph, table, formula, and Hebrew-English text.

### Reviewer Agent

Reviews the manuscript for clarity, consistency, assignment coverage, and course alignment. It produces a review report and corrected manuscript.

### LaTeX Agent

Creates a LaTeX assembly specification. It does not compile the PDF. It prepares rendering intent for deterministic Python code.

## 7. Data Flow

```text
config/
  -> generated/intermediate/book_plan.json
  -> generated/intermediate/research_pack.json
  -> generated/intermediate/manuscript.md
  -> generated/intermediate/review_report.json
  -> generated/intermediate/latex_spec.json
  -> references.bib
  -> agent_pipeline_graph.png
  -> validation_report.json
  -> generated LaTeX files
  -> final.pdf
```

Committed example files use the `sample_` prefix under `data/intermediate/`. Dry-run refreshes those examples into `generated/intermediate/` on each run and never calls `crew.kickoff()`. A real CrewAI run (`--run-crew`) is implemented as an opt-in, API-key-guarded path: it writes task outputs to the same generated runtime directory, records `real_run_trace.json`, captures token usage when CrewAI exposes it, and emits budget alerts from `config/budgets.json`. Real outputs must pass content-depth gates before replacing canonical artifacts; shallow output is preserved under `generated/intermediate/real_raw/`. The delivered manuscript content remains authored deterministically so the default path costs nothing.

## 12. How To Run The Project

The CLI entry point is `uv run --no-project --with pydantic --with matplotlib --with jinja2 python -m bookgen.main --dry-run [--build-pdf] [--run-crew]`, also exposed as the `bookgen` console script. Dry-run is the default and never calls the API; `--run-crew` requires `OPENAI_API_KEY`, routes through the gatekeeper, and persists real-run evidence when deliberately enabled.

```powershell
bookgen --dry-run
bookgen --dry-run --build-pdf
```

Run the current local startup check:

```powershell
cd <path-to>\AI-Agent-Orchestration-HW3
$env:PYTHONPATH="src"
uv run --no-project --with pydantic --with matplotlib --with jinja2 python -m bookgen.main
```

Explicit dry-run:

```powershell
uv run --no-project --with pydantic --with matplotlib --with jinja2 python -m bookgen.main --dry-run
```

Run unit tests:

```powershell
$env:PYTHONPATH="src"
uv run --no-project --with pydantic --with pytest --with matplotlib --with jinja2 python -m pytest tests/unit
```

Generate deterministic graph and bibliography:

```powershell
$env:PYTHONPATH="src"
uv run --no-project --with matplotlib python -c "from bookgen.harness.graph_generator import generate_agent_pipeline_graph; from bookgen.harness.citations import generate_references_bib; generate_agent_pipeline_graph(); generate_references_bib()"
```

## Guidance For Future AI Assistants

Before editing code, read this file, `docs/IMPLEMENTATION_STATUS.md`, and `docs/COURSE_ALIGNMENT.md`.

Respect these boundaries:

- Do not add extra agents beyond the approved five unless explicitly requested.
- Do not call any API unless the user explicitly approves an API milestone.
- Do not compile PDF before the renderer/compiler milestone.
- Keep deterministic components deterministic.
- Update implementation status after each milestone.
